# **S1-選課系統測試文件：新增課程(Create Course)**

**文件版本：** V1.0  
**文件日期：** 2025/12/08  
**專案作者：** 張育鴻


## **1. 進入條件 (Preconditions)**

在執行「新增課程」功能的測試前，需滿足以下條件：

1. **測試環境建置：**  
   - 前後端已部署於測試伺服器 (Staging)  
   - API `/course/create`、`/auth/login` 已可正常使用  

2. **教師帳號：**  
   - 資料庫中已存在至少一組教師帳號  
   - 教師帳號具備「新增課程」權限  

3. **系統狀態：**  
   - 系統可正常登入  
   - 資料庫可正常寫入課程資料  

4. **基礎資料已建立：**  
   - 系統內已有至少一個可選的「上課地點」  
   - 課程命名不得重複（若系統有此限制）

---

## **2. 測試方法 (Test Procedure)**

本功能採用 **黑箱測試 (Black-box Testing)**，測試者不需了解後端邏輯，只需透過 UI 或 API 驗證是否正確。

測試流程包含：

1. **依照測試案例逐步操作 UI 介面或呼叫 API**  
2. **比對畫面與 API 回應是否符合預期結果**  
3. **必要時查詢資料庫 `course` 資料表，確認寫入的欄位是否正確**  
4. **若結果不符，回報為 BUG 並標註嚴重程度**

---

## **3. 測試成功條件 (Success Criteria)**

1. **所有測試案例通過率達 100%**  
2. **無 Critical 或 Major 等級的系統缺陷**  
3. **所有案例覆蓋「新增課程需求規格」中定義的 AC1～AC6**

---

## **4. 新增課程任務拆解 (Task Breakdown)**

### **Task：新增課程 (Create Course)**

| 流程步驟 | 說明 |
|--------|------|
| **T1** | 教師登入系統並通過身份認證 |
| **T2** | 進入「新增課程」頁面 |
| **T3** | 輸入課程基本資料：<br>‧ 課程名稱<br>‧ 上課時間（週別、開始/結束時間）<br>‧ 上課地點<br>‧ 人數上限<br>‧ 選課截止日期 |
| **T4** | 系統驗證輸入資料格式（日期格式、時間區間、上限人數等） |
| **T5** | 驗證課程是否與已存在課程重複（若系統有此規則） |
| **T6** | 儲存課程資料至資料庫 |
| **T7** | 顯示「新增成功」並回到課程列表頁 |

## **1\. 進入條件 (Preconditions)**

在執行測試案例之前，必須滿足以下條件：

1. **測試環境建置:** 系統程式碼已部署至測試伺服器 (Staging Environment)，且資料庫已連接。  
2. **基礎資料準備:** 資料庫中需預先建立測試用的基礎資料，包含：  
   * 數間不同容量的教室（例如：E-201 容量50人, A-101 容量80人）。  
   * 多位測試用的教師帳號與登入 Token。  
   * 部分已排定的課程資料（用於測試衝突情境）。  
3. **系統狀態:** 系統處於可運行狀態，API 服務正常回應，前端頁面可正常載入。

## **2\. 測試方法 (Test Procedure)**

本測試採用 **黑箱測試 (Black-box Testing)** 方法，測試人員無需了解程式碼內部邏輯，僅需透過使用者介面 (UI) 與 API 進行驗證。

1. **步驟執行:** 依照測試案例中的「步驟」逐一操作系統。  
2. **結果比對:** 觀察系統回應與畫面顯示，比對是否符合「預期結果」。  
3. **資料驗證:** 必要時需查詢後端資料庫，確認資料寫入與更新的正確性（如：確認教室 ID 是否正確寫入課程資料表）。  
4. **缺陷紀錄:** 若實際結果與預期不符，需記錄為 Bug，並標註嚴重程度。

## **3\. 測試成功條件 (Success Criteria)**

本次測試活動視為成功的標準如下：

1. **案例通過率:** 所有定義的測試案例 (Total Test Cases) 通過率需達 **100%**。  
2. **無重大缺陷:** 測試過程中未發現 **Severity 1 (Critical)** 或 **Severity 2 (Major)** 等級的缺陷。  
3. **需求覆蓋:** 測試案例必須覆蓋需求規格書中定義的所有驗收標準 (AC1 \~ AC5)。

# **S8-選課系統測試文件：排課管理與衝突偵測系統**

# **1. 測試目的**

本測試文件旨在驗證「排課管理與衝突偵測系統」模組是否符合需求文件之功能需求與非功能需求。

核心功能驗證： 確保教務行政人員能夠正確執行排課數據查詢、排程調整（含衝突解決）以及選課規則配置。

關鍵非功能驗證： 確保系統能滿足衝突偵測效率（效能）、資料交易原子性（可靠性）和權限存取控制（安全性）要求。

---

# **2. 測試範圍**

本測試涵蓋以下主功能：

**課程與教室使用狀況查閱：** 驗證管理員可正確查詢所有課程資訊和教室的時段佔用情況，並使用多條件篩選。

**排程調整與衝突調解**：驗證系統在調整排程時能根據**外部衝突警示**提供**調解建議**，並根據規則**阻止**不合法的排程保存。

**選課規則與時程管理：** 驗證管理員可正確配置**選課起訖時間**和**分階段選課規則**。

**系統限制與資料一致性：** 驗證系統在排程修改時能確保數據交易的完整性，並強制限制未解決衝突的課表發佈。
