# **1. 測試目的**

本測試文件旨在驗證「排課管理與衝突偵測系統」模組是否符合需求文件之功能需求與非功能需求。

核心功能驗證： 確保教務行政人員能夠正確執行排課數據查詢、排程調整（含衝突解決）以及選課規則配置。

關鍵非功能驗證： 確保系統能滿足衝突偵測效率（效能）、資料交易原子性（可靠性）和權限存取控制（安全性）要求。

---

# **2. 測試範圍**

本測試涵蓋以下主功能：

**課程與教室使用狀況查閱：** 驗證管理員可正確查詢所有課程資訊和教室的時段佔用情況，並使用多條件篩選。

**排程調整與衝突調解**：驗證系統在調整排程時能根據**外部衝突警示**提供**調解建議**，並根據規則**阻止**不合法的排程保存。

**選課規則與時程管理：** 驗證管理員可正確配置**選課起訖時間**和**分階段選課規則**。

**系統限制與資料一致性：** 驗證系統在排程修改時能確保數據交易的完整性，並強制限制未解決衝突的課表發佈。

# **4. 測試案例（Test Cases）**


| Case ID | 測試目標 (Test Goal) | 步驟 (Steps) | 預期結果 (Expected Result) |
| :--- | :--- | :--- | :--- |
| TC-8.1.1 | 驗證多條件複合篩選的準確性 | 1. 管理員登入並進入課程列表。<br>2. 同時輸入課程名稱與教師姓名進行篩選。 | 頁面在 3 秒內載入，並只顯示符合所有篩選條件的課程。 |

| TC-8.2.2 | 驗證無衝突排程修改與數據保存 | 1. 選擇一門無衝突課程。<br>2. 修改其教室為空閒教室。<br>3. 保存後查詢新教室排程。 | 修改成功保存，新教室顯示已被該課程佔用。 |
| TC-8.3.1 | 驗證分階段選課規則的配置與儲存 | 1. 進入選課規則設定頁。<br>2. 設定「第一階段」僅限「大四」學生。 | 規則成功儲存，並於學生端啟用限制邏輯。 |
| TC-8.3.2 | 驗證非開放時間前端提示 | 1. 以學生身分，在非選課開放時間點擊選課按鈕。 | 顯示：「尚未開放選課」或「選課已截止」。 |
| TC-8.5.1 | 驗證資料交易原子性（高併發） | 1. 多名管理員於同一時間嘗試修改同一課程排程。 | 只有一次修改成功寫入，資料無部分更新或遺失。 |

# **5. 預期結果**

每個測試案例的執行結果必須完全符合其在 **Section 4** 描述的預期結果 (Expected Result)。

任何與預期結果不符的現象，將被記錄為缺陷 **(Defect)**，並提交給開發團隊進行修正。特別是涉及**衝突偵測**和**資料完整性**的案例，實施零容忍原則。
